# Flutter vs Kotlin 学习/复习模块迁移审查报告

## 1. 审查范围

- Flutter 新项目：`e:\Nemo\packages\features\feature_learning\lib\src`
- Kotlin 旧项目：`e:\Nemo\old_nemo\Nemo\feature\learning\src\main\java\com\jian\nemo\feature\learning`
- 对比对象：
  - 学习模块（学习会话、SRS 评分流、头部交互、队列调度）
  - 复习模块（复习准备页、复习会话、评分后流转、等待机制）

## 2. 总体结论

结论：**Flutter 项目尚未完整还原 Kotlin 项目的学习模块与复习模块（UI 与业务逻辑均存在明显缺口）**。

- 学习模块：基础流程已迁移（加载队列→显示答案→评分→更新进度），但高级能力缺失较多（撤销、等待态、suspend/bury、自动朗读开关、答案延迟、会话恢复策略完整度等）。
- 复习模块：UI 外形有一定还原，但业务逻辑差距较大；目前实现更接近“简化版复习”，未达到 Kotlin 版“严格调度 + 重学状态机 + 钉子户治理”。
- 综合判断：**当前还原度约 55%（学习约 65%，复习约 40%）**。

## 3. 学习模块对比

### 3.1 已还原部分（Flutter 已具备）

1. 基础学习主流程
   - 有学习页、卡片翻页、显示答案、四档评分按钮、进度条。
   - Flutter 证据：`learning_screen.dart`、`learning_providers.dart`、`srs_action_area.dart`。

2. SRS 基础调度能力
   - 有学习步进 + FSRS 毕业计算 + 间隔预览。
   - Flutter 证据：`srs_scheduler.dart`。

3. 新学与到期项混合入队（按 mode）
   - Flutter 会先拉取到期项，再补当日新项配额。
   - Flutter 证据：`learning_repository.dart`。

### 3.2 未还原/降级点（关键差异）

1. 头部菜单行为未接通
   - Kotlin 头部菜单支持：撤销评分、评分说明、Suspend、Bury、自动朗读开关、显示答案等待开关/时长。
   - Flutter 头部虽有“更多”图标，但菜单点击逻辑为空。
   - 影响：交互能力与策略控制明显缺失。

2. 撤销（Undo）机制缺失
   - Kotlin 有 UndoSnapshot + 回滚 DB + 回滚队列/统计。
   - Flutter 无对应事件/状态机。

3. 等待态（Waiting）与 Learn Ahead 策略缺失
   - Kotlin 有 `LearningQueueManager`，可返回 Next/Wait/Empty，并支持 `resumeFromWaiting`。
   - Flutter 仅“当前项评分后前进/重排”，无等待态与强制恢复流程。

4. 卡片状态标记未完整还原
   - Kotlin：NEW/REVIEW/RELEARN 动态计算。
   - Flutter 学习页固定传 `CardBadge.fresh`，未体现复习/重学状态变化。

5. 音频与跟读练习接线不完整
   - Kotlin 学习卡完整接入 onSpeakWord/onSpeakExample/onPracticeClick。
   - Flutter 卡片组件支持这些回调，但学习主页面未传入，主流程里不可用。

6. 会话恢复与会话持久化能力不对齐
   - Kotlin 有 `SessionLoader` + 保存会话 ids/index/steps/waitingUntil 的恢复流程。
   - Flutter 当前学习态由 Provider 内存维护，缺少等价的完整会话恢复链路。

## 4. 复习模块对比

### 4.1 已还原部分（Flutter 已具备）

1. 有独立复习页面与评分流程
   - 可显示卡片、显示答案、四档评分、完成页。
   - Flutter 证据：`review_screen.dart`、`review_providers.dart`。

2. 复用学习卡片 UI 体系
   - 词卡/语法卡视觉一致性较好，基础体验接近 Kotlin 版。

### 4.2 未还原/逻辑偏差（核心问题）

1. 复习队列来源错误（重大）
   - Kotlin 复习仅拉取 `due words + due grammars`。
   - Flutter 复习调用 `getLearningQueue(mode)`，该方法会混入“新项配额”，导致复习会话掺入新内容。
   - 这是业务语义偏差，不是单纯 UI 差异。

2. 复习维度被简化为单 mode
   - Kotlin 复习会话是单词+语法统一混排（按到期策略）。
   - Flutter 复习以 `mode` 运行，无法对齐 Kotlin 统一复习体验。

3. 重学状态机与钉子户治理缺失
   - Kotlin 复习有：Again 触发惩罚、重学 step、Leech 阈值、skip/bury_today 策略、严格队列重排。
   - Flutter 复习评分后仅做简单前进，不维护等价的重学状态机。

4. 等待机制缺失
   - Kotlin 复习有 Waiting 状态与恢复入口。
   - Flutter 无 Waiting UI 与时序控制。

5. 复习准备页数据源未接真实业务（重大）
   - Flutter `session_prep_providers.dart` 使用 mock 数据。
   - Kotlin 复习准备页由 ViewModel 实时加载到期内容，并在 ON_RESUME 自动刷新。

6. 复习入口路由参数存在风险
   - Flutter 复习路由定义为 `review/:mode`，但准备页按钮 `pushNamed(LearningRouteNames.review)` 未传 `pathParameters`。
   - 可能导致路由跳转异常/参数缺失。

## 5. UI 还原度评估

1. 学习主界面外观
   - 顶部结构、卡片、底部评分区域大体接近。
   - 但高级菜单与交互状态（undo/suspend/bury/等待）未跟上，属于“样子像，能力不全”。

2. 复习准备与复习会话界面
   - 样式方向接近，但流程结构仍偏简化。
   - Kotlin 是“准备页→复习会话（严格调度）”，Flutter 当前实现未完整对齐。

## 6. 业务逻辑还原度评估

1. 学习模块
   - 基础 SRS 流程可用。
   - 但 Kotlin 的关键策略能力（恢复、等待、撤销、Leech/手动处置）未完整迁移。

2. 复习模块
   - 与 Kotlin 逻辑差距较大，是当前主要短板。
   - 特别是“队列来源”“统一混排”“重学状态机”三个核心点未对齐。

## 7. 优先级整改建议（按影响排序）

P0（必须先修）
1. 复习队列改为“仅 due 项”，禁止混入新项。
2. 复习入口修复路由参数传递问题。
3. 复习准备页改为真实数据源（替换 mock）。

P1（高优）
4. 复习改为词法统一混排队列，并补齐 Waiting/Resume。
5. 学习模块补齐头部菜单动作（Undo/Suspend/Bury/延迟显示/自动朗读）。
6. 学习页接入 onSpeakWord/onSpeakExample/onPracticeClick。

P2（中优）
7. 会话持久化与恢复能力对齐 Kotlin（含 waitingUntil、steps、index）。
8. 学习卡片 badge 动态化（新学/复习/重学）。

## 8. 最终判定

**Flutter 项目目前未完全还原 Kotlin 项目的学习模块与复习模块。**

- UI：有较高相似度，但交互控制项缺失较多。
- 业务：学习模块“可用但降级”，复习模块“核心逻辑未对齐”。
- 建议优先完成 P0/P1 后再做一次复审，以确认是否达到“功能等价迁移”标准。
