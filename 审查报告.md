# Nemo Flutter 项目架构审查报告

## 1. 审查概览

- 审查时间：2026-03-26
- 审查范围：`apps/nemo_app`、`packages/core/*`、`packages/features/*`、`melos.yaml`
- 审查目标：围绕「解耦与可维护性、团队协作、复用与插件化、按需加载」四个维度，评估现状并给出可落地的升级方案

### 1.1 总体结论

当前项目已经具备 **Monorepo + Flutter Package 模块化雏形**，是一个正确方向的基础架构；但与“最先进架构”相比，仍处于 **中期过渡阶段**：

- 优点：已使用 workspace（Melos）+ 多 package 拆分 + GoRouter 模块路由暴露
- 主要问题：模块边界不够严格、跨 Feature 直接依赖、App 层仍保留本地功能页、测试深度不足、Deferred Loading 尚未落地

综合评级（当前）：**B-（可演进，但需系统化治理）**

---

## 2. 现状审查（按你的四项要求）

## 2.1 解耦与可维护性

### 达成项

- 已形成 `apps + packages` 的工程结构，主应用通过 path dependency 引入多个模块
- 已拆分 core/feature 包，例如：
  - `packages/core/core_designsystem`
  - `packages/core/core_ui`
  - `packages/core/core_domain`
  - `packages/features/feature_learning` 等
- 主路由已经通过各 feature 暴露的 route 聚合（GoRouter），具备模块入口统一编排能力

### 不足项

- App 层仍保留 `apps/nemo_app/lib/features/*` 本地页面，与 `packages/features/*` 并存，存在边界重叠
- Feature 间存在直接耦合，不是只依赖 core/shared 契约：
  - `feature_learning` 直接依赖 `feature_library`
  - `feature_statistics` 直接依赖 `feature_learning`、`feature_library`
- 当前 core 仅有 UI/Domain，缺少强约束的 `core_network / core_data / core_auth / core_storage` 等跨项目复用基础模块

### 结论

模块化方向正确，但尚未形成“严格分层 + 可替换模块”的治理闭环；当单模块问题出现时，仍有较大概率牵连其他 Feature。

---

## 2.2 团队协作

### 达成项

- 已使用 `melos.yaml` 管理 workspace（`apps/**` + `packages/**`）
- 模块已 package 化，理论上支持多人并行开发和独立发布节奏

### 不足项

- workspace scripts 非常少，缺少标准化流水线脚本（如 analyze/test/format/coverage/graph）
- 缺少模块依赖方向自动校验（无 import 规则、无依赖图守卫）
- 测试主要是 smoke/export 级别，尚未形成“模块独立可回归”的质量门禁体系

### 结论

协作基础具备，但仍偏“约定式协作”，缺少“工具化强约束协作”。

---

## 2.3 复用与插件化

### 达成项

- 已将 UI 主题、通用组件、领域模型拆分至 core 包
- Feature 入口以 package 暴露，具备跨 app 复用前提

### 不足项

- Feature 内仍有 UI + 业务 + 路由 + 跨 feature 跳转耦合，复用颗粒度不够干净
- 关键横切能力尚未沉淀为可复用插件模块（网络、鉴权、数据访问、缓存、埋点、远程配置）
- 缺少“接口包（contract）+ 实现包（impl）”机制，导致替换成本较高

### 结论

可复用基础存在，但距离“插件化生态可扩展”仍有明显差距。

---

## 2.4 按需加载（Deferred Loading）

### 现状

- 项目中未发现 `deferred as` / `loadLibrary()` / DeferredWidget 等实现
- 当前路由聚合方式是静态 import，所有 feature 在启动构建时即被打入主包

### 影响

- 大型应用场景下首包体积和首屏加载压力会持续增长
- 难以按场景（低频功能）做冷启动优化

### 结论

Deferred Loading 当前为 **未落地状态**，这是与“先进大型 Flutter 架构”差距最大的点之一。

---

## 3. 关键问题清单（高优先级）

1. 模块边界不严：App 内 feature 与 package feature 双轨并存
2. Feature 横向依赖过多：违背“feature 只依赖 core/shared”
3. 缺少统一的数据/网络/鉴权核心模块
4. 缺少依赖方向自动化约束（防止架构退化）
5. 缺少高价值模块测试（金丝雀用例、契约测试、集成测试）
6. Deferred Loading 未启用
7. workspace 脚本体系不完整，不利于 CI 门禁
8. 架构文档与 Flutter 实际结构存在代际混杂（仓库仍包含旧 Android 模块目录）

---

## 4. “最先进 Flutter 架构”目标蓝图（建议）

## 4.1 推荐分层与模块边界

建议统一成以下结构（保持 Monorepo，不另起仓）：

- `apps/nemo_app`：壳工程（仅组合、路由装配、环境注入）
- `packages/core/*`：跨业务核心能力  
  - `core_designsystem`（主题/组件）  
  - `core_domain`（实体/值对象/业务规则）  
  - `core_network`（Dio/Chopper 或 gRPC client + 拦截器）  
  - `core_storage`（Hive/Isar/SharedPreferences 封装）  
  - `core_auth`（身份态、token 刷新、权限）  
  - `core_observability`（日志、埋点、崩溃、性能）  
- `packages/shared/*`：跨 feature 的可复用业务片段（非核心基础设施）
- `packages/features/*`：按业务域拆分（每个 feature 内部保持 clean-ish 分层）

依赖方向必须单向：

`app -> feature -> shared -> core`  
`feature -> feature`（禁止，改为依赖 shared/contract）

---

## 4.2 Feature 内部结构建议

每个 feature 包建议标准化：

- `presentation/`（UI、路由适配）
- `application/`（状态管理、用例编排）
- `domain/`（feature 内部业务规则，可依赖 core_domain）
- `infrastructure/`（repository impl、datasource）
- `public_api.dart`（仅暴露必要入口）

这样能做到：

- UI 改版不影响数据实现
- 替换后端或存储时仅改 infrastructure
- 可在 feature 维度做独立单测、集成测试

---

## 4.3 依赖治理与协作治理

建议补齐以下工程化能力：

- Melos 脚本标准化：
  - `melos run analyze`
  - `melos run test`
  - `melos run format`
  - `melos run verify`（串联 analyze + test + 依赖规则校验）
- 引入依赖边界检查：
  - 自定义 lint（禁止 feature 互相 import）
  - CI 中校验 `import` 规则
- 版本与发布策略：
  - feature/core package 采用语义化版本
  - 变更日志自动生成（按 package）

## 5. Deferred Loading 落地方案（可执行）

## 5.1 Web 端（立即可做）

对低频功能包采用 deferred import：

- 示例思路：测试模块、统计高级页面、设置中的低频子页
- 路由首次进入时执行 `loadLibrary()`，再渲染页面
- 配合骨架屏/加载态组件，避免白屏感知

适用场景：Flutter Web、超大多页面应用。

---

## 5.2 Android 端（进阶）

采用 Flutter Deferred Components（Play Feature Delivery）：

- 将低频业务拆为动态特性组件
- 与路由和资源分发策略协同
- 通过 A/B 对比冷启动时间、安装包体积、首屏内存峰值

适用场景：Android 大型包体、需要显著降低初始下载成本的场景。

---

## 5.3 优先级建议

1. 先完成“代码层 deferred import + 路由加载态”
2. 再评估 Android Deferred Components 的工程成本与收益
3. 最后纳入性能基线看板（包体/启动/帧率/内存）

---

## 6. 技术栈升级建议（对标 2026 主流）

在保留你当前技术选型优势（Riverpod + GoRouter + Melos）的前提下，建议升级到以下组合：

- 状态管理：Riverpod（保留）+ codegen 全面规范化
- 路由：GoRouter（保留）+ typed route（统一参数契约）
- 网络：Dio（或 Chopper）+ OpenAPI 代码生成
- 数据：Isar/Hive（离线优先）+ repository 抽象
- 观测：Sentry/Firebase Crashlytics + 性能埋点
- 质量：Very Good Analysis 或自定义 lints（含边界规则）
- 自动化：Melos + CI matrix（按 package 并行 analyze/test）

---

## 7. 分阶段整改路线图

## Phase 1（架构收敛）

- 移除或迁移 `apps/nemo_app/lib/features/*` 到对应 feature package
- 禁止 feature 互相直接依赖，抽离到 `shared` 或 `core contract`
- 新增 `core_network / core_storage / core_auth` 基础模块骨架

## Phase 2（工程治理）

- 完成 melos scripts 与 CI 门禁
- 增加模块级单测、契约测试、路由集成测试
- 引入依赖边界 lint

## Phase 3（性能与加载）

- 对低频 feature 实施 deferred import
- 建立性能基线：包体、TTI、首帧、内存峰值
- 评估并落地 Android Deferred Components（如收益显著）

---

## 8. 验收标准（建议）

达到以下标准，可认为升级完成：

- 模块依赖图无 feature 横向依赖
- App 壳层仅做装配，不承载业务页面实现
- 每个 feature 至少具备：路由集成测试 + 核心业务单测
- CI 对每次 PR 执行模块化校验（边界 + analyze + test）
- 至少 2 个低频模块完成 deferred loading 并有性能收益数据

---

## 9. 最终评价

你的项目已经拥有良好的现代化基础，但要满足“最先进架构和技术栈”的目标，还需要从“已拆包”升级到“强边界 + 强治理 + 强性能”的工程体系。  
按本报告路线推进后，可以实现：

- 单模块故障可控、回归范围可控
- 团队并行开发冲突显著下降
- 核心能力可跨项目复制
- 大型应用场景下启动与包体得到系统优化
